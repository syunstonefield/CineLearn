# decide — CineLearn 意思決定モード

重要な判断を「一人の直感で即答」ではなく、**独立した複数視点で発散 → 互いに叩いて弱点を潰す → 実データで裏取りして収束**という型で下すための作業モード。Opus 4.8 の生の能力を、構造で取りこぼしなく引き出すのが狙い。

このスキルは新しい能力を足すものではなく、**すでに CineLearn で有効だった型**（3部署討論・サブエージェント報告の裏取り・実データ主義）を毎回サボらず回すための強制装置。元になった運用知見はメモリ `dept-debate-protocol` / `verify-subagent-claims` / `debug-with-real-data` にある。

## いつ使うか（判断の重さで決める）

**使う**：価格・収益化設計、機能の取捨、ローンチ戦略、法務にかかる判断、アーキテクチャ選定、コード監査、「A と B どちらにすべきか」、間違えるとデプロイ/課金/信頼のコストが大きい判断、答えの幅が広く一人だと盲点が出そうな問い。

**使わない**：事実の一問一答、軽微なコード編集、会話的なやり取り。ここで討論を回すのはトークンの無駄（下記コスト意識）。

司会は常にメインループ（あなた自身）。サブエージェントは意見を出す参加者であって、決めるのはメイン。

## コスト意識（先に釘を刺す）

各ラウンド・各参加者はサブエージェント数ぶんトークンが N 倍になる（実測=3部署1ラウンドで約115k）。だから：
- ラウンドは固定せず**難度で 1〜3 に可変**（下記）。
- 多エージェント分離は効く場面だけ on-demand で発動、常設化しない。
- 「議論で解ける対立」は収束させ、「価値観の分岐」は即ユーザーへエスカレ（何ラウンド回しても解けない）。

## ワークフロー

### 0. スコープ確定（討論を回す前に必ず）
- 論点を分解し、**何を決めるのか**を1〜2文で固定する。
- **既に確定した前提を洗い出し、「蒸し返し禁止」として参加者に明示的に渡す**（例：freemium一択・DRM不可侵・¥680確定 など）。これをやらないと参加者が決着済みの議論を再燃させ、ラウンドが浪費される。
- 難度を見積もってラウンド数を決める：
  - 小さい判断 → **1パス**（各視点のポジション＋メインの統合のみ・反論なし）
  - 普通 → **2R**（＋相互反論。R2 後に収束判定）
  - 紛糾・重大 → **3R 上限**（収束 or 分岐確定で終了。4R 以上は回さない）

### 1. R1＝発散（独立・並列）
- プロジェクトのサブエージェント（`marketing` / `user-voice` / `engineer` / `legal` / `security` など問いに合うもの）を**並列で**起動。
- 各参加者には**他部署の見解を見せない**（アンカリング＝集団思考を防ぐ）。同じスコープ資料・前提・確定事項だけを渡す。
- 各回答は「結論→根拠2〜3点→リスク1点」の型を要求すると統合しやすい。分量は 600〜800字程度に絞らせる。
- 重大・答えの幅が広い問いでは**同一部署の多エージェント分離**を提案してよい（パターンB＝独立インスタンス並列）。例：`user-voice` を[学習者/視聴者/競合経験者]の3体、`engineer` を[MVP最速]対[完成度・保守]、`legal` を[最大限保守]＋[悪魔の代弁者＝どこまで攻められるか]。1エージェントに複数役を順に演じさせる（パターンA）は文脈が混ざって意見が丸まるので重大判断では避ける。

### 2. R2＝敵対（相互レビュー）
- 各参加者に**他者の見解の要旨**を `SendMessage`（完了済みエージェントは agentId で再開）で渡し、**一番弱い所を叩かせる**。言い換え・追認は禁止、反論か自説更新を要求。
- 「自説を変える点／絶対に譲らない点」を明示させると収束点が見える。
- **ここでメインが収束判定する**（実務を動かすのは主に R2）。

### 3. 収束判定（R2 後に必ず）
- (a) 収束した → R3 で合意を確定して打ち切り。
- (b) 事実・技術でまだ割れる（議論で解ける）→ R3 へ。
- (c) 価値観・リスク選好の分岐に煮詰まった（何R回しても解けない）→ **即打ち切り、ユーザーに決定を要求**。討論を続けない。

### 4. R3＝収束・記録
- メインが統合し、「確定した決定」「根拠」「保留＝要ユーザー判断」を分けて書く。
- 結論は `docs/` に記録し、`backlog`（メモリ）の該当項目を更新する（秘書運用）。価格等の重要決定は**3部署一致でも最終承認はユーザー**である旨を添える。

## 実データ・実コードでの裏取り（討論と並ぶ二本目の柱）

討論で出た主張・サブエージェントの報告を、**確定・記録・反映する前に必ず一次情報で検証する**。これを飛ばすと、もっともらしいが誤った結論がそのまま docs や本番に乗る。

- **サブエージェントの「実装済み／検証済み」は実ディスクで裏取り**：`git status --short` / `ls <新規ファイル>` / `grep <キーワード>` / 構文チェック。過去に engineer が「curl 12/12 PASS」と報告したのに4ファイルが存在しなかった実例がある（`verify-subagent-claims`）。デプロイ・コミット等の不可逆操作の前は必須。
- **バグ・数値・挙動は推測でなく実データで確定**：仮説が1回外れたら2回目を推測で出さず、実機の console 出力・実際の値・runtime 挙動・`curl` 単体検証を取りに行く（`debug-with-real-data`）。Disney+ 例文バグは推測修正4連敗→実 `document.title` 1つで即確定した。
- **env はpullで確認できない**→ runtime 挙動で検証（`infra-gotchas`）。
- **本番反映の確認は実 curl で**：git push やコミットステータスは cinelearn-next 本番の反映根拠にならない（`vercel deploy --prod` が必要）。「反映済み」と言う前に本番 HTML/挙動を実際に叩いて確かめる。
- ⚠ 長いシークレット（JWT等）をチャットでユーザーに貼らせない（UIがマスクしてコピーが壊れる）。掃除・修復はアプリ内自動修復として実装する。

討論の結論に「まだ実データで裏取りしていない主張」が残っているなら、それは**ドラフト**であって確定ではない。裏取りできるまで docs にそう明記する。

## CineLearn 固有の前提（薄く常に被せる）

判断はこのプロジェクトの憲法の内側で行う。これらに反する結論は、どれだけ魅力的でも採らない。

- **法務ガードレール**：DRM 不可侵・再配布不可。例文＝著作権法32条の引用＋48条の出所明示。全文キャッシュ＝30条の4（情報解析）で非配信。Netflix 内部APIシークは承認済みの技術的例外で他サービスへ広げない。OS 商用契約はフェーズ2。詳細は `public-launch-legal-posture`。
- **設計の憲法**：視聴は常に自由・無摩擦。**罪悪感UI禁止**（「開きたい」はOK、「開かないと後ろめたい」はNG）。変動報酬やオープンループを課金の人質にしない。
- **効果主張は研究裏付けの範囲のみ**（「効果2倍」「確実に定着」は禁止）。
- **非提携スタンス**：Netflix/Amazon/Disney/TMDB/OpenSubtitles と提携・公認関係にない。名称・商標は各権利者に帰属。
- **秘書運用**：重要な決定・保留・発見は `backlog` に記録し、可逆でない/外向きの操作（デプロイ・push・公開・ストア操作）は必ずユーザー承認を取る。

## この型が効く理由

独立サンプリング（R1で他者を見せない）は相関する誤りを減らし、単一文脈では見えない相違・盲点を出す（多部署討論で `/api/example` の設計欠陥が浮上したのと同じ原理）。敵対レビュー（R2）は「もっともらしいが弱い主張」を殺す。実データ裏取りは「賢いが間違った結論」が確定・本番反映されるのを止める。3つ揃って初めて「手を抜かない判断」になる。
