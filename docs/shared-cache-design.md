# 共有単語キャッシュ ＋ 厳選カタログ：要件定義・設計書

- 対象機能: 生成済み単語データのサーバー共有キャッシュ／初期は厳選作品ホワイトリスト＋事前シード
- 文書ステータス: ドラフト（実装着手前のレビュー用）
- 関連メモ: リリース実行ロードマップ（フェーズ0関門「キャッシュ」）、公開方針と規約/著作権リスク

---

## 1. 背景・目的

### 背景
- 現状、同一の「作品×シーズン×話」を別ユーザーが学習するたびに、毎回 Claude API（Haiku 4.5）でフル生成している。
- 1話あたりのコストは概算 **約 $0.05（入力～10.6k / 出力～7.8k トークン）**。コストが「ユーザー数 × 視聴本数」に線形に比例し、青天井になりうる。
- さらに OpenSubtitles の取得クォータ（匿名5/ログイン20 件/日）があり、サーバーキャッシュ未実装のままだと即パンクする。

### 目的
1. **コストの定数化**: 生成は「ユニークな話数」ぶんだけにし、ユーザー数に依存させない。
2. **クォータ枯渇の回避**: 字幕取得もキャッシュ／事前取得で日次制限を回避する。
3. **初期品質の安定**: 公開前に厳選作品を事前生成し、内容を確認（QC）した状態で提供する。
4. **法的防御の単純化**: こちらが管理する作品範囲に限定し、出所明示と引用範囲の制御をやりやすくする。

### 非目的（このフェーズではやらない）
- 任意作品の即時オープン生成（フェーズ2以降）。
- ユーザー個別データ（履歴・SRS・マイ単語）のクラウド同期方式の変更（既存のまま）。
- 字幕全文のユーザー向け表示（引用1文に限定する方針は維持）。

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| 共有キャッシュ | 生成済み単語データをサーバー（Supabase/Postgres）に保存し、全ユーザーで共有する仕組み |
| カタログ（ホワイトリスト） | 初期に利用を許可する厳選作品の一覧（TMDB ID で管理） |
| シード（warming） | 公開前にカタログ作品を一括生成し、キャッシュに事前投入する作業 |
| キャッシュキー | キャッシュ1件を一意に識別する文字列。`cache_version + 正規化タイトル + season + episode` |
| `cache_version` | 生成ロジック（プロンプト/モデル）の版。上げると新版で再生成・上書きされる |

## 3. 要件定義

### 3.1 機能要件（FR）

| ID | 要件 |
|---|---|
| FR-1 | 単語生成の前に共有キャッシュを参照し、ヒットすればキャッシュの単語データを返す（生成しない） |
| FR-2 | キャッシュキーは `cache_version + 正規化タイトル(englishTitle優先) + season + episode` で構成する |
| FR-3 | 初期フェーズは「カタログに含まれる作品」のみ生成・表示の対象とし、カタログ外は「近日対応」表示にする |
| FR-4 | カタログ作品のキャッシュは、事前シードスクリプトで一括投入できる |
| FR-5 | キャッシュには単語データ本体に加え、出所メタ（字幕プロバイダ、取得日時、`cache_version`、モデル名）を保存する |
| FR-6 | `cache_version` を上げた場合、旧版キャッシュは参照されず、再生成によって新版が作られる |
| FR-7 | クイズは引き続きテスト実行時にのみ生成する（遅延生成。実装済み） |
| FR-8 | ユーザー向けに字幕を表示する場合は1文（引用）に限定し、全文を表示しない |

### 3.2 非機能要件（NFR）

| ID | 要件 |
|---|---|
| NFR-1（コスト） | カタログN作品×話数を1回シードした後は、ユーザー数が増えても追加生成コストが発生しないこと |
| NFR-2（性能） | キャッシュヒット時の単語表示は生成待ちゼロ（DB読み取りのみ）であること |
| NFR-3（保存量） | 1話あたり数十KBのテキストJSON想定。1万話規模でも数百MBに収まり、Supabase無料枠で運用可能 |
| NFR-4（堅牢性） | キャッシュ参照/書き込みが失敗しても、アプリは従来フロー（都度生成）に安全にフォールバックすること |
| NFR-5（セキュリティ） | 一般ユーザー（anon）はキャッシュを**読み取りのみ**。書き込みは特権ロール（サーバー側）に限定し、キャッシュ汚染を防ぐこと |
| NFR-6（永続性） | キャッシュは明示削除/バージョン更新まで永続。腐敗防止のため `cache_version` による無効化手段を持つこと |

### 3.3 法的要件（公開方針メモと整合）

| ID | 要件 |
|---|---|
| LR-1 | 字幕全文のサーバー保存は「情報解析目的の内部運用」（著作権法30条の4）の枠で行い、外部にそのまま再配布・全文表示しない |
| LR-2 | ユーザーに見せる例文は引用（32条）の範囲（1文・出所明示）に限定する |
| LR-3 | 出所（OpenSubtitles 等）をデータに保持し、必要に応じて明示できるようにする（48条 出所明示） |

## 4. 設計

### 4.1 全体アーキテクチャ

```
[クライアント(Next.js)]
   │ 1. 作品×S×E を指定して単語要求
   ▼
[API: /api/vocab  （新規・サーバー側）]
   │ 2. 共有キャッシュ参照（SELECT, service_roleで読む or anon read）
   ├─ ヒット ───────────────► 単語JSONを返す（生成なし）
   │
   └─ ミス（カタログ内のみ許可）
        │ 3. 字幕取得（/api/subtitles, 取得済みなら再利用）
        │ 4. Claude生成（/api/claude, server側）
        │ 5. refineDramaWords等で整形
        │ 6. キャッシュ書き戻し（INSERT, service_role）
        ▼
        単語JSONを返す

[シードスクリプト（手元/CI, service_role）]
   カタログを走査 → 3〜6を一括実行してキャッシュを事前投入
```

> 重要な設計判断: **生成と書き込みはサーバー側（特権ロール）に寄せる**。クライアントは anon キーしか持たず、これは公開鍵なので、クライアントから直接キャッシュへ書かせると汚染リスクがある（NFR-5）。初期フェーズではユーザー起点の書き込み経路を持たず、シードスクリプトのみが書き込む構成が最も安全。

### 4.2 データモデル（Supabase / Postgres）

> ⚠️ **重要：2つのキャッシュを1テーブルに混ぜない（決定事項A）。**
> 「生成出力（単語＋1文例文＝32条引用の世界・ユーザー配信OK）」と「字幕全文（30条の4・情報解析・非配信）」は法的根拠も配信可否も保持方針も異なる。**別テーブル**にする。

#### テーブル①: `vocab_cache`（生成出力／32条・配信OK・version無効化で永続）
| カラム | 型 | 説明 |
|---|---|---|
| `id` | uuid (PK, default gen_random_uuid()) | 主キー |
| `cache_key` | text (UNIQUE, NOT NULL) | `v{n}:tmdb{id}:s{S}e{E}`（**TMDB ID 基準**。決定事項B） |
| `cache_version` | int (NOT NULL) | 生成ロジックの版 |
| `tmdb_id` | int (NOT NULL) | 作品の TMDB ID（一次キー要素・カタログ照合） |
| `season` | int (NOT NULL) | |
| `episode` | int (NOT NULL) | |
| `display_title` | text | 表示用の原題（任意） |
| `title_norm` | text | 正規化タイトル（補助・デバッグ用。キーには使わない） |
| `words` | jsonb (NOT NULL) | 生成済み単語配列（後述スキーマ） |
| `word_count` | int | 単語数（運用・品質ゲート用） |
| `subtitle_provider` | text | 出所（例: `opensubtitles`） |
| `model` | text | 生成モデル（例: `claude-haiku-4-5`） |
| `created_at` | timestamptz (default now()) | |
| `updated_at` | timestamptz (default now()) | |

- インデックス: `UNIQUE(cache_key)`、補助に `(tmdb_id, season, episode, cache_version)`。

#### テーブル②: `subtitle_raw_cache`（字幕全文／30条の4・**非配信**・TTL必須）
クリック保存（経路②→①）で任意単語の元文を引くには ~50語の `vocab_cache` では足りず、字幕全文が要る。全文は性質が違うため分離する。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | uuid (PK) | 主キー |
| `tmdb_id` | int (NOT NULL) | |
| `season` | int (NOT NULL) | |
| `episode` | int (NOT NULL) | |
| `provider` | text | 出所（例: `opensubtitles`）＋ file_id |
| `srt` | text (NOT NULL) | 字幕全文（**生SRTをクライアントへ返さない**） |
| `expires_at` | timestamptz (NOT NULL) | **TTL**（例: 取得後30日）。溜め込まない |
| `created_at` | timestamptz (default now()) | |

- インデックス: `UNIQUE(tmdb_id, season, episode)`。期限切れは定期削除（または `expires_at` 判定で無視＋遅延GC）。
- **配信制約**: このテーブルの `srt` はサーバー内部の情報解析（単語生成・クリック単語の文抽出）にのみ使用。クライアントへは**該当1文（引用）だけ**を返す（LR-2）。
- 代替案（より安全・transient）: 全文をサーバー保存せず「観ている回の全文を拡張側で一時保持」する。保持期間が視聴中に限られるぶん法的フットプリントは最小。**初期はこの代替案 or 短TTL のどちらかを選ぶ**。

#### `words` (jsonb) の要素スキーマ（既存 `generateVocab` 出力に準拠）
```json
{
  "word": "原形",
  "level": "A2|B1|B2|C1|C2",
  "pos": "名詞|動詞|形容詞|副詞",
  "definition": "日本語の意味",
  "example": "字幕由来の英文（drama）/作文（plus）",
  "example_ja": "exampleの日本語訳",
  "tier": "core|advanced|context",
  "source": "drama|plus"
}
```

#### テーブル: `catalog`（ホワイトリスト）
| カラム | 型 | 説明 |
|---|---|---|
| `tmdb_id` | int (PK) | 作品の TMDB ID |
| `title_norm` | text | 正規化タイトル |
| `display_title` | text | 表示名 |
| `type` | text | `tv` / `movie` |
| `enabled` | bool (default true) | 公開フラグ |
| `seasons` | jsonb | 対応シーズン/話数の指定（任意） |

### 4.3 キャッシュキー設計（決定事項B：TMDB ID 基準）
- キーは**タイトルではなく TMDB ID** を使う。理由：タイトルはローカライズ／表記揺れ（"The Office" vs "Office"）／リメイクで衝突・破綻する。カタログが TMDB ID ホワイトリストであり、`api/subtitles.js` も既に tmdb_id 検索を使っているため、ID 基準が堅い。
- 共有キャッシュキー形式: `v{cache_version}:tmdb{tmdb_id}:s{season}e{episode}`
  - 例: `v1:tmdb66732:s1e1`（Stranger Things）
- `title_norm` は補助フィールド（デバッグ/表示）としてのみ保持し、**キーには使わない**。
- `cache_version` をキーに含めることで、版を上げるだけで旧キャッシュが自然に未ヒット化し、再生成→新版作成に流れる（FR-6/NFR-6）。

### 4.4 処理フロー（擬似コード）

```
function getVocab(drama, season, episode):
    title_norm = normalize(drama.englishTitle || drama.title)
    # 初期フェーズ: カタログ外は弾く
    if not catalog.isEnabled(drama.tmdb_id): 
        return { blocked: true }   # UIは「近日対応」表示
    key = `v${CACHE_VERSION}:${title_norm}:s${season}e${episode}`
    hit = vocab_cache.selectByKey(key)
    if hit: return hit.words                      # FR-1: 生成せず返す
    # ミス時（サーバー側のみ・特権ロール）
    raw = subtitles.fetchOrReuse(title_norm, season, episode)
    words = claude.generateVocab(raw, ...)        # 既存ロジック
    words = refineDramaWords(words, raw)
    vocab_cache.upsert(key, words, meta)          # FR-5 メタ付与, service_role
    return words
```

- フォールバック（NFR-4）: キャッシュ参照や書き込みで例外が出ても、生成結果はその場でユーザーに返す（キャッシュは「あれば速い」最適化として扱い、必須依存にしない）。

### 4.5 シード（warming）スクリプト
- 入力: `catalog` テーブル（または JSON 定義）。
- 処理: 各作品の対応話を順に `getVocab` 相当で生成→`vocab_cache` に upsert。
- 実行: 手元（service_role キーを環境変数で渡す）で1回／カタログ追加時に随時。
- **OpenSubtitles クォータ計画（決定事項C）**: シード自体が DL 上限を消費する。
  - 無料ログイン = 20件/日 → 100話シードに5日。
  - 個人 VIP（$15/年）= 1000件/日 → 100話を1日で投入可。**無料＝非商用の段階は VIP 正規利用が現実的**。
  - これは「有界カタログ＋継続関係」での**正規な事前取得**であり、以前棄却した harvest-and-abandon（無差別収集→放棄）とは別物。シード速度を上限内に収める設計とする。
  - 24h以内の同一 file 再取得はクォータ非消費。失敗リトライはこれを利用。
- Claude コスト見積: 例）20作品×10話=200話 × $0.05 ≒ **$10 一度きり**。

### 4.6 表示制限ガード（法的）
- 単語データはサーバー保存するが、UI で出すのは `example`（1文・引用）と出所のみ。
- 字幕全文（`cl_sub_raw_*` 相当）はサーバー内部解析用途に留め、画面に全文を出さない実装を維持（LR-1/LR-2）。

### 4.7 既存コードとの統合ポイント
| 既存 | 変更/追加 |
|---|---|
| `next-app/lib/supabase.js` | 読み取り専用方針は維持。共有キャッシュ用に**別テーブル**を新設（本番データに非干渉） |
| `next-app/lib/vocab.js` `generateVocab` | ロジックは流用。呼び出しをサーバーAPI経由に寄せる（生成＋書き込みをサーバー側へ） |
| `api/`（Vercel Functions） | `api/vocab`（キャッシュ参照→ミス時生成→書き戻し）を新規追加 |
| `next-app/lib/subtitles.js` キャッシュキー | タイトル正規化規則を共有キャッシュと共通化 |
| サービス選択/作品追加 UI | カタログ外を「近日対応」にする分岐を追加（FR-3） |

## 5. セキュリティ

- **RLS（Row Level Security）を有効化**する（決定事項D：公開前に実機確認）。
  - `vocab_cache` / `catalog`: anon ロールは `SELECT` のみ許可。`INSERT/UPDATE/DELETE` は禁止。
  - `subtitle_raw_cache`: anon は **SELECT も不可**（非配信。サーバー service_role のみ読み書き）。
  - 書き込みは **service_role**（サーバー側＝シードスクリプト／`api/vocab`）のみ。service_role キーはクライアントに絶対に出さず、サーバー環境変数で保持。
- anon キーは公開前提（クライアント同梱。実際 `js/supabase.js` / `extension/background.js:5` にハードコードされ誰でも読める）だが、RLS により読み取り限定なので汚染・破壊は不可。
- **★GRANT 必須（実機検証 2026-06-16）**: 本番は anon にテーブル GRANT 自体が無く、未認証 SELECT は RLS 以前に **401 `permission denied for table`（42501）** になる（既存5テーブルで確認）。よって `vocab_cache`/`catalog` を公開読みさせるには **RLS ポリシーだけでなく `GRANT SELECT ... TO anon, authenticated;` を明示**する必要がある（`supabase_shared_cache.sql` に反映済み）。`subtitle_raw_cache` は anon に GRANT しない（非配信）。
- **書き込みモデルが2種類共存することに注意**：
  - 共有キャッシュ（`vocab_cache` / `subtitle_raw_cache`）= **サーバー専用書込**（service_role）。
  - 個人データ（既存 `my_words` 等）= **ユーザー書込**（anonキー＋`user_id` RLS）。
  - 設計上、**`my_words` を新しいキャッシュ経路に通さない**こと（書込権限モデルを取り違えない）。
- service_role キーは**リポジトリに含めない**（`.env`、Vercel 環境変数で管理）。

## 6. 運用

- **無効化/再生成**: `cache_version` をインクリメント → 旧キャッシュは未ヒット化。必要作品のみシードで作り直す。
- **監視**: `vocab_cache` の件数・`word_count` 分布をたまに確認（空/極小は生成失敗の兆候）。
- **保存量**: テキストJSONのため軽量。肥大時は古い `cache_version` 行を一括 DELETE。

## 7. リリース段階（ロードマップ整合）

| フェーズ | 内容 |
|---|---|
| フェーズ0（公開前） | 共有キャッシュ＋カタログ＋事前シードを実装。RLS設定。引用表示ガード。**シードのみが書き込む保守スタート**。 |
| フェーズ1（無料運用） | カタログ限定で公開。ユーザーが見た話を自動キャッシュ追加（サーバー書き込み経路を追加）。**品質ゲートを通す（決定事項E）** |
| フェーズ2（収益化後） | 任意作品の生成を有料/上限付きで開放 |

> **決定事項E：フェーズ1の品質ゲート。** 自動生成はサーバー書込なので汚染（偽データ注入）は防げるが、品質は無レビューで全員に配信され、悪い生成が `cache_version` を上げるまで残る。フェーズ1で自動追加を開放する際は、キャッシュ書込前に簡単なゲートを通す：
> - 語数が下限以上か（`word_count`）
> - `drama` の `example` に対象 `word`（活用形）が含まれるか
> - 既存の検査ロジック（`js/app.js:2936` 周辺 / `refineDramaWords`）を流用
> ゲート不合格は書き込まず、その回はキャッシュ化を見送る（フォールバック生成のみ返す）。

## 8. 決定事項（レビュー反映済み）

| ID | 決定 |
|---|---|
| A | 字幕全文は `vocab_cache` と**別テーブル `subtitle_raw_cache`** に分離。根拠=30条の4・**非配信**・**TTL必須**・溜め込まない。代替として「拡張側で視聴中のみ一時保持（transient）」も可。`vocab_cache` は32条（引用1文）の世界として配信OK・version永続。 |
| B | キャッシュキーは**TMDB ID 基準**（`v{n}:tmdb{id}:s{S}e{E}`）。title はキーに使わず補助フィールドに留める。 |
| C | シードは OpenSubtitles クォータを消費。VIP（1000/日）で正規取得、上限内速度で設計。harvest-and-abandon とは区別。 |
| D | `vocab_cache`/`catalog`=anon SELECTのみ・service_role書込、`subtitle_raw_cache`=anonアクセス不可。RLSを公開前に実機確認。`my_words`（ユーザー書込）と書込モデルを混同しない。 |
| E | フェーズ1の自動追加は品質ゲート（語数・例文整合）を通してから書き込む。フェーズ0はシードのみの保守スタート。 |

## 8.1 初期カタログ方針（推奨）
- **狭く深く始める**（DL量・品質レビュー・法的フットプリントを抑える）。
- 例：**人気＋セリフが学習向き**な作品を **5〜10本 × 各シーズン** で開始 → フェーズ1で自動拡張。
- ジャンルは**専門語が映える法廷／医療／ビジネス系**（例：Suits 的なビジネス/法務）。**TOEIC連動という差別化**に合致し、初期のフックが立つ。

## 8.3 確定設計（2026-06-16・上記を正とする）

- **インフラ = A（リーン）**: テーブルは `vocab_cache` と `catalog` の 2 つのみ。DDL は `supabase_shared_cache.sql`。`subtitle_raw_cache`・サーバー生成は追加（足すだけ）で後付け可能＝将来性を損なわない。
- **レベル別生成 = スーパーセット＋読み取り時絞り**:
  - 生成（シード時）は 1 話につき **広め（CEFR A2〜C2 を網羅・語数多め）に 1 回だけ**。各語の `level`/`tier` を正直にタグ付けして `words` に保存。`refineDramaWords`（drama は字幕に実在＋逐語例文）と `example_ja` 補完まではシード時に行う。
  - 表示（読み取り時）は学習者の TOEIC/CEFR で **バンド絞り＋除外語（`getExcludeSet`）＋語数キャップ＋tier 選択** を当て込む。AI 呼び出しゼロ。レベル変更時も即・再最適化。
  - → **コストは 1 話 = 1 回のまま（真の定数化）**。`cache_key` は `v{n}:tmdb{id}:s{S}e{E}`（**レベルを含めない**＝決定事項B 維持）。
  - DB に持つ「レベル情報」= 各語の `level`/`tier`（既存スキーマ）＋行の `coverage_min`/`coverage_max`（このスーパーセットがカバーするバンド範囲）。**学習者個人のスコアは持たない**（profiles 側で読み取り時適用）。
  - 実装影響: [generateVocab](../next-app/lib/vocab.js) を「生成段（広く生成・字幕突合・和訳補完）」と「表示段（除外・バンド絞り・キャップ）」に割る軽い改修が要る。
